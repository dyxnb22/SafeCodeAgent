# v2.2.5 — Subagent Result Merge Policy

## Summary

Adds a deterministic merge policy for subagent findings so `AgentLoop` can safely incorporate read-only investigation results into the main session context without losing provenance.

## New Module

`src/safecode/subagents/merge_policy.py`

### Types

**`SubagentFinding`** (frozen dataclass):

| Field | Type | Description |
|---|---|---|
| `task_id` | `str` | Unique task identifier from the dispatched subagent |
| `summary` | `str` | High-level summary of findings |
| `observations` | `list[str]` | Ordered observation strings |
| `files_inspected` | `list[str]` | File paths read during investigation |
| `errors` | `list[str]` | Error messages (populated on failure) |
| `blocked` | `bool` | Whether execution was blocked |
| `success` | `bool` | Whether the investigation succeeded |

**`MergedSubagentContext`** (frozen dataclass):

| Field | Type | Description |
|---|---|---|
| `summary` | `str` | Joined summaries from successful findings |
| `observations` | `list[str]` | Deduplicated, capped observations |
| `files_inspected` | `list[str]` | Deduplicated, capped file paths |
| `source_task_ids` | `list[str]` | Task IDs of successful findings |
| `blocked_task_ids` | `list[str]` | Task IDs of failed/blocked findings |
| `errors` | `list[str]` | Error strings from failed/blocked findings |

### Function

**`merge_subagent_findings(findings, max_observations=20, max_files=50)`**

Merge behavior:
- Input order is preserved throughout.
- Only successful findings (`success=True` and not `blocked`) contribute to `summary`, `observations`, and `files_inspected`.
- Observations and files are deduplicated; first occurrence wins.
- `max_observations` and `max_files` caps are applied after dedup (deterministic: first N entries).
- Errors and `blocked_task_ids` are collected from failed/blocked findings.
- Empty task IDs are skipped to avoid polluting provenance lists.
- Never raises on empty input, empty fields, or all-blocked findings.

## Compatibility

- `SubagentResult` fields (`task_id`, `summary`, `observations`, `files_inspected`, `errors`, `success`, `blocked`) map directly to `SubagentFinding` fields. Callers can convert without transforming data.
- `AgentLoop._execute_subagent_dispatch_step()` journal payload keys already match `SubagentFinding` fields — no breaking changes to the journal schema.
- Execution behavior is unchanged: `SubagentDispatchExecutor` and `AgentLoop` are not modified.
- Empty `task_id` (returned by `_fail("")` before task creation) is silently skipped in provenance lists — this is intentional to avoid storing empty-string IDs.

- `TestMergeSubagentFindingsEmpty` — empty input produces a valid empty context without raising

- `TestMergeSubagentFindingsDedupe` — first-occurrence deduplication for observations and files
- `TestMergeSubagentFindingsCaps` — max_observations and max_files caps enforced deterministically
- `TestMergeSubagentFindingsBlocked` — blocked/failed findings excluded from summary; errors/blocked_task_ids collected
- `TestMergeSubagentFindingsEdgeCases` — empty fields, single finding, all-blocked
- `TestMergeFromSubagentResult` — integration: `SubagentResult` → `SubagentFinding` → `merge_subagent_findings`
