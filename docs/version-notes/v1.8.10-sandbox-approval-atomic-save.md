# v1.8.10 Release Notes — Sandbox Approval Atomic Save

## Summary

v1.8.10 hardens sandbox execution approval persistence. Approval creation,
legacy consume, and execution claim updates now share one atomic JSON write
path based on same-directory temporary files and `os.replace(...)`.

## What Changed

### Approval Atomic Writes
- Added `SandboxExecutionApprovalStore._atomic_write_json(...)`.
- `approve(...)` now writes approval JSON through a hidden random temp file and
  atomically replaces the final approval path.
- `consume(...)` uses the same helper for legacy single-use updates.
- `claim_for_execution(...)` keeps its lock-file guard, but now uses the shared
  random-temp atomic write helper instead of a fixed `.tmp` file.

### Symlink Safety
- Saving an approval over an existing same-name symlink replaces the symlink
  path itself.
- The symlink target is not modified.

## Security
- Reduces risk of partial approval JSON after interrupted writes.
- Avoids direct writes through same-name symlinks in the user-level approval
  directory.
- Keeps the existing approval trust boundary: approval roots inside the project
  are still rejected.

## Backward Compatibility
- Approval JSON schema is unchanged.
- Approval path hashing is unchanged.
- `is_approved(...)`, `load_approval(...)`, `approve(...)`, `consume(...)`,
  `claim_for_execution(...)`, and `revoke(...)` keep the same public behavior.
