# v1.8.9 Release Notes — Sandbox Proposal Atomic Save

## Summary

v1.8.9 hardens pending sandbox execution proposal persistence. Proposal JSON is
now written through a same-directory temporary file and published with
`os.replace(...)`.

## What Changed

### Atomic Proposal Save
- `SandboxExecutionProposalStore._write(...)` writes JSON to a hidden temp file
  in `.sac/`.
- It then atomically replaces `pending_sandbox_execution.json` with
  `os.replace(...)`.
- Leftover temp files are cleaned up if publishing fails.

### Symlink Safety
- Creating a proposal over a broken same-name symlink replaces the symlink path
  itself.
- The external symlink target is not created or modified.

## Security
- Reduces risk of partial pending proposal JSON after interrupted writes.
- Avoids direct writes through broken same-name symlinks in the project-local
  `.sac/` directory.
- Keeps the existing protections for already-present valid or corrupt pending
  proposal files: new proposals are still refused until the existing pending
  path is discarded.

## Backward Compatibility
- Pending proposal schema is unchanged.
- `create(...)`, `load_pending()`, `discard_pending()`, `propose(...)`,
  `approve(...)`, `preflight`, and `execute` keep the same public behavior.
