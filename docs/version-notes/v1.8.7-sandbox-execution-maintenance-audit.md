# v1.8.7 Release Notes — Sandbox Execution Maintenance Audit

## Summary

v1.8.7 closes the audit loop for sandbox execution result maintenance.
Confirmed result pruning now writes a durable audit event, while dry-run
pruning remains preview-only.

## What Changed

### CLI Audit Event
- `sac sandbox executions prune --keep-latest N --yes` writes
  `sandbox_execution_results_pruned`.
- The event records pruning parameters and counts:
  - `keep_latest`
  - `status_filter`
  - `candidate_count`
  - `deleted_count`
  - `remaining_count`
  - capped `proposal_ids`
  - `proposal_ids_truncated`

### Dry-Run Behavior
- `sac sandbox executions prune --dry-run` does not write the destructive
  maintenance audit event.

## Security
- Audit metadata does not include stdout, stderr, full command argv, or
  command output.
- Proposal IDs are capped to the first 20 candidates to avoid unbounded audit
  metadata growth.
- Existing v1.8.6 prune safety guards are unchanged: symlinks, corrupt JSON,
  non-result files, and mismatched source paths are still protected.

## Backward Compatibility
- `SandboxExecutionResultStore.prune()` still returns an integer deletion
  count.
- `sac sandbox executions prune --dry-run` and `--yes` keep the same CLI
  flags and output shape.
