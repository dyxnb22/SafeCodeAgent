# v1.8.4 Release Notes — Sandbox Execution Audit & Usability

## Summary

v1.8.4 adds schema versioning to result records, forward-compatible reading,
filtering on `sac sandbox executions`, and execution result summary in
`sac sandbox status`.

## What Changed

### Schema Version
- Every result record JSON now includes `_schema_version: "v1"` for future
  migration detection.
- Records written by older versions (without the field) still load correctly.

### Robust Reads
- `load()` and `list_all()` now tolerate extra unknown JSON keys (forward
  compatibility) while still requiring all known required fields.
- Corrupt individual result files are silently skipped by `list_all()`.
- `_filter_record_data()` strips unknown keys before constructing the
  `SandboxExecutionResultRecord` dataclass.

### CLI Filtering
- `sac sandbox executions` now accepts:
  - `--status completed|blocked_claim`
  - `--backend none|macos_seatbelt|linux_bubblewrap|docker`
  - `--proposal-id <substring>`
- Underlying API: `SandboxExecutionResultStore.filter_by()`.

### Status Summary
- `sac sandbox status` now includes an "Execution Results" panel showing
  total/completed/blocked counts plus the latest result summary.

## Security
- No changes to redaction/truncation rules in v1.8.3.
- No new fields expose command content or env values.
- `_schema_version` is metadata-only, not security-sensitive.

## Backward Compatibility
- All existing CLI commands unchanged.
- `SandboxExecutionResultRecord` frozen dataclass unchanged.
- Old result records load without modification.
