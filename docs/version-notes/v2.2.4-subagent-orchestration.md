# v2.2.4 — Subagent Orchestration

## Summary

Adds a safe, read-only subagent orchestration layer so the main agent can delegate bounded investigation tasks to subagents and merge their findings back into the main loop without allowing writes or unapproved command execution.

## New Components

### `src/safecode/subagents/executor.py`

- `SubagentRequest` — frozen dataclass with `task`, `scope`, `max_steps`.
- `SubagentResult` — frozen dataclass with `task_id`, `summary`, `observations`, `files_inspected`, `commands_attempted`, `blocked_actions`, `errors`, `success`, `blocked`.
- `SubagentDispatchExecutor` — validates args via `ToolCallAdapter`, enforces `max_steps` bounds (1–10), delegates to `ReadonlySubagentRunner`, parses result into structured `SubagentResult`. Never raises — all failure paths return a blocked result.

### `src/safecode/tools/registry.py`

Added `subagent.dispatch` spec:
- Risk: `low`, Permission: `subagent`, `requires_human_approval=False`.
- Required args: `task` (str), `scope` (str), `max_steps` (int).
- Audit event: `subagent_dispatched`.

### `src/safecode/state/journal.py`

- Added `"subagent_dispatch"` to `JournalEventType`.
- Added `record_subagent_dispatch(session_id, step, message, dispatch_summary)`.

### `src/safecode/agent/tools.py`

- `ToolIntentRouter` now routes `subagent` intents to `subagent.dispatch` (was `subagent.inspect`).
- `executable_now=True`, no human approval required — strictly read-only.

### `src/safecode/agent/loop.py`

- Routes `subagent.dispatch` intents to `_execute_subagent_dispatch_step()`.
- `_execute_subagent_dispatch_step()` extracts task/scope/max_steps from `ToolIntent`, calls `SubagentDispatchExecutor`, records a `subagent_dispatch` journal event with the full structured result payload.

## Safety Properties

- Subagents must not write files outside `.sac/subagents/` — enforced by `ReadonlySubagentRunner` + `FilesystemBoundary`.
- `readonly=True` enforced on every task; non-readonly tasks are blocked before execution.
- No MCP write tools or shell commands are executed by subagents.
- All failures fail closed and return structured `SubagentResult` — never raise through the loop.
- Subagent results are merged into the main plan as observations via the journal, not as a separate uncontrolled session.

## Compatibility

- Existing `subagent.run_readonly` and `subagent.inspect` registry entries are unchanged.
- MCP read/write routing is unchanged.
- All existing journal event types are preserved; `subagent_dispatch` is append-only.
- `SubagentDispatchExecutor.execute()` never raises.
- `max_steps` is clamped and validated: values outside `[1, 10]` return a blocked result.
