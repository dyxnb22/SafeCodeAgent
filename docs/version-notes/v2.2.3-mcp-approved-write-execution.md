# v2.2.3 Notes: MCP Approved Write Execution

## Summary
The bounded agent loop can now execute MCP write tools that have been explicitly approved through the proposal flow, while maintaining the existing safety boundary for unapproved or rejected proposals.

## What Changed

**`src/safecode/mcp/proposal.py`**
- `MCPWriteProposalStore.approve_pending(proposal_id)` — verifies proposal_id matches and status is "pending", then marks the proposal "approved" and persists.
- `MCPWriteProposalStore.reject_pending(proposal_id)` — verifies proposal_id matches, marks the proposal "rejected" and persists. Raises `PermissionError` on mismatch or missing proposal.

**`src/safecode/mcp/runner.py`**
- `MCPReadOnlyRunner.execute_approved_write(server, tool, input_data, trace_id)` — executes a write tool without a read-only classification gate. Enforces all other policies (server config, command policy, network, filesystem, input/output size limits). Audits with `mcp_approved_write_started` / `mcp_approved_write_completed` event types. Returns a blocked `MCPRunResult` for all infrastructure failures (never raises).

**`src/safecode/mcp/loop_executor.py`**
- `MCPApprovedWriteExecutor` — mirrors `MCPReadToolExecutor` structure. Validates via `ToolCallAdapter` (`mcp.propose_write` spec), checks the stored proposal is "approved" and matches server/tool, calls `execute_approved_write` on the runner, discards the proposal after execution (regardless of outcome). Fails closed for every failure path: bad format, no proposal, wrong status, proposal_id mismatch, server/tool mismatch, runner error. Never raises.

**`src/safecode/agent/loop.py`**
- `AgentLoop._find_approved_write_proposal(tool_name)` — loads the pending proposal and returns it only if status is "approved" and server/tool match.
- `AgentLoop._execute_mcp_approved_write_step(plan_item, state, routed, proposal_id)` — calls `MCPApprovedWriteExecutor.execute()`, updates session state, journals an `mcp_call` event with `approved_write=True` in the payload.
- `AgentLoop.step()` — after the existing `mcp.call_readonly` branch, checks for an approved write proposal matching the current write intent. If found, routes to `_execute_mcp_approved_write_step`; otherwise falls through to the existing `mcp.propose` pending_action path.

## Safety Properties
- Write tools still route to `mcp.propose` initially — the proposal gate is not bypassed.
- Execution requires the proposal to have status "approved"; "pending" or "rejected" proposals are always blocked.
- Proposal_id verification prevents stale or cross-session proposal re-use.
- Server/tool mismatch between the intent and the proposal is blocked.
- Proposal is discarded after execution (success or failure) to prevent replay.
- `MCPApprovedWriteExecutor` never raises — all failure paths produce a blocked observation.
- Network, filesystem, shell, and input/output size policies are enforced by `execute_approved_write` (same infrastructure as read-only).
- Read-only MCP path (`MCPReadToolExecutor`) is unchanged.

## Compatibility
- Existing `mcp.propose` routing and write/unknown intent behavior are unchanged.
- All v2.2.2 MCP read-loop tests continue to pass.
- `JournalEventType` and `record_mcp_call()` are unchanged; the new approved-write path reuses them with `approved_write=True` in the payload.
- CLI commands `sac mcp propose-write`, `sac mcp pending`, `sac mcp discard` are unchanged.
