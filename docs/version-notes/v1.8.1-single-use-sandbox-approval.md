# v1.8.1: Single-Use Sandbox Execution Approval

## Summary

Makes sandbox execution approvals **single-use**. After a `sac sandbox execute`
succeeds, the associated approval is consumed and cannot pass preflight a
second time. Blocked preflight (e.g. unsupported backend, command policy
violation) does NOT consume the approval.

## Changes

### Source
- `src/safecode/sandbox/approvals.py` — `SandboxExecutionApproval` gains a
  `consumed: bool = False` field. `is_approved()` returns `False` when
  consumed. New `consume()` method marks an approval as consumed.
  Backward-compatible with v1.8.0 approval files (missing field → not consumed).
- `src/safecode/sandbox/execution.py` — `execute_pending()` calls `consume()`
  after a successful execution and writes `sandbox_execution_approval_consumed`
  audit event.

### Tests
- `tests/test_sandbox_execution_security_evals.py` — 7 new tests in
  `TestSingleUseApproval`: second-exec blocked, blocked preflight doesn't
  consume, consumption audit event, `is_approved()` after consumption, legacy
  file backward compat, no env leaks, unsupported backend no consumption.

### Safety Invariants
All preserved from v1.8.0:
- Proposal integrity (hash verification)
- Approval binding (user-level, outside project root, TTL enforced)
- Command policy (dangerous commands blocked even with approval)
- Network policy (disabled-by-default enforced)
- Filesystem boundary (no escape from project root)
- Audit (every execution + consumption logged)
- No OS sandbox binary invoked
