# v1.8.5 Release Notes — Sandbox Execution Detail + Query Polish

## Summary

v1.8.5 adds a detail view for individual execution records, limit/sort
controls on the list command, and testable query parameters on the
underlying `filter_by()` method.

## What Changed

### New CLI: `sac sandbox execution show <proposal-id>`
- Full metadata table: proposal ID, attempted time, backend, executed,
  exit code, duration, status, command head, hash prefix, message,
  output lengths.
- stdout/stderr preview panels when available.
- Non-zero exit (1) with clear error when proposal ID is not found.

### Enhanced `sac sandbox executions`
- `--limit N`: cap output to N records.
- `--sort newest|oldest`: change sort order (newest is default).
- Invalid limit or sort values are caught and reported as errors.

### Extended `filter_by()` API
- New params: `sort_order` (default `"newest"`) and `limit` (default `None`).
- Raises `ValueError` on invalid values so callers get clear validation.
- Default behavior matches v1.8.4 — no breaking changes.

## Security
- No changes to redaction/truncation rules.
- No new fields in result records.
- All existing safety invariants preserved.

## Backward Compatibility
- `filter_by()` defaults unchanged.
- `sac sandbox executions` without new flags produces same output.
- All existing CLI commands preserved.
