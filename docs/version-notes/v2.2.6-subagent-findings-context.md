# v2.2.6 — Subagent Findings Context Integration

## Summary

Wires the v2.2.5 merge policy into the main agent loop so successful `subagent_dispatch` journal events are converted into `SubagentFinding`, merged via `merge_subagent_findings()`, and exposed to the agent context passed to `choose_tool` during planning/tool selection.

## New Module

**`src/safecode/subagents/journal_adapter.py`** — adapter with three public functions:
- `_event_to_finding(event)` — converts one `subagent_dispatch` `AgentJournalEvent` payload to `SubagentFinding`; returns `None` on malformed payloads, never raises.
- `findings_from_journal_events(events)` — filters `subagent_dispatch` events and converts each to `SubagentFinding`; silently skips malformed payloads.
- `merge_journal_subagent_findings(events, max_observations=20, max_files=50)` — converts and merges in one call; never raises.

## AgentLoop Changes

`AgentLoop.step()` now calls `_enrich_with_subagent_findings(session_id, context)` between `ContextCollector.collect()` and `llm_client.choose_tool()`. If the session journal contains any prior `subagent_dispatch` events with successful, failed, or blocked outcomes, the merged result is injected into the context dict as `"subagent_findings"` with keys: `summary`, `observations`, `files_inspected`, `source_task_ids`, `blocked_task_ids`, `errors`.

## Safety Rules

- Failed/blocked subagents contribute only `errors` and `blocked_task_ids`, not `summary`, `observations`, or `files_inspected`.
- Malformed journal payloads (non-dict, missing keys, wrong types for list fields) are silently skipped — the agent loop never crashes.
- `_enrich_with_subagent_findings()` catches all exceptions and returns context unchanged if any error occurs (fail closed).
- No execution behavior change: only context enrichment; `SubagentDispatchExecutor` and `ReadonlySubagentRunner` are unaffected.
- Deterministic ordering and max_observations/max_files caps from v2.2.5 are preserved.
