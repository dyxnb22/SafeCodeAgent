# v2.3.6 — Agent Loop Patch Path

## Summary

`AgentLoop` can now produce pending patch proposals and stop for approval.
`sac agent run "fix calculator bug"` generates `.sac/pending_patch.json` via the
existing `AgentOrchestrator.edit()` workflow and stops with
`stopped_reason == "approval_required"` before any business/source file is touched.

## What changed

**`src/safecode/agent/loop.py`**
- Imports `AgentOrchestrator` for patch proposal delegation.
- `step()` now routes `patch.propose` intents through `_execute_patch_proposal_step()`.
- `_execute_patch_proposal_step()` delegates to `AgentOrchestrator.edit(goal)`,
  saves `.sac/pending_patch.json`, updates session status to `waiting_for_user`, and
  returns `stopped_for_approval=True`. Fails closed: if a pending patch already exists
  or if proposal generation raises, no business files are modified.

**`src/safecode/llm/mock.py`**
- `choose_tool()` returns a `patch` intent when the goal contains "calculator",
  enabling deterministic test coverage of the patch proposal path.
- `propose_patch()` returns the calculator repair patch when task contains "calculator",
  independent of context file listing.

**`src/safecode/state/journal.py`**
- Added `"patch_proposed"` to `JournalEventType`.
- Added `record_patch_proposal()` to record proposal details (patch_id, files, scope).

**`src/safecode/cli_agent.py`**
- `sac agent run` now prints a highlighted **Approval Required — Pending Patch** panel
  when `stopped_reason == "approval_required"` and the pending action is a patch proposal.
  The panel shows the patch file path, patch ID, target files, and next-step commands.

## Compatibility

- Legacy `sac edit`, `sac apply`, preview/apply behavior is unchanged.
- Read-only agent runs (non-write goals) continue to use the `context.read` path.
- MCP and subagent dispatch are unchanged.
- `AgentOrchestrator.edit()` is called unchanged from `loop.py`; no duplication of
  patch parsing, validation, diff building, audit logging, or pending patch serialization.
- Fail-closed invariants: the agent only writes `.sac/pending_patch.json` through
  `AgentOrchestrator.edit()`; it never directly modifies business/source files.
  If a pending patch exists, the agent stops for approval without overwriting.

## Honest capability statement

v2.3.6 connects the `sac agent run` path to the existing patch proposal workflow.
The agent can now propose a patch and pause for human review — it is not a fully
autonomous coding agent. Patch application still requires explicit `sac apply` approval.
