# v2.2.1 — Model Tool Call Adapter

## What Changed

**New: `src/safecode/tools/adapter.py`**
- `AdapterError` — ValueError subclass raised for all validation failures.
- `ToolCallValidationResult` — frozen Pydantic model with `tool_name`, `spec`,
  `resolved_args`, `requires_approval`, `risk`, `permission_category`, `audit_event`.
- `ToolCallAdapter` — validates LLM tool intents against the v2.2.0 `ToolRegistry`:
  - `lookup(tool_name)` — name lookup; raises `AdapterError` for unknowns.
  - `validate(tool_name, args)` — full validation (name + required args + types);
    returns `ToolCallValidationResult` or raises `AdapterError`.
- No execution, no LLM calls, no file writes.

**Updated: `src/safecode/agent/tools.py`**
- `ToolIntentRouter` now imports and uses `ToolCallAdapter` for every routing decision.
- A `_REGISTRY_NAMES` mapping resolves each `ToolIntentType` to its registry tool name.
- `requires_human_approval` from `ToolSpec` is combined with the intent's own flag
  (more restrictive wins) — WRITE/SHELL/high-risk tools always require approval.
- Unknown registry names raise `ValueError` (fail closed).
- All public route strings, `RoutedToolIntent` fields, and CLI behavior are unchanged.

## Compatibility
- `ToolCallAdapter` and `ToolCallValidationResult` are new public surface — callers
  must not mutate returned results (frozen Pydantic).
- `AdapterError` is a `ValueError` subclass — existing `except ValueError` handlers
  will catch it without change.
- `ToolIntentRouter.route()` behavior is backward compatible for all valid known intents.
