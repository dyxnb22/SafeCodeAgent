# v1.8.6 Release Notes — Sandbox Execution Result Maintenance

## Summary

v1.8.6 adds safe maintenance operations for the execution result store:
aggregate statistics, dry-run prune preview, and confirmed prune with
symlink/path-safety guards.

## What Changed

### `SandboxExecutionResultStats`
- New frozen dataclass with: `total_records`, `completed_count`,
  `blocked_claim_count`, `oldest_attempted_at`, `newest_attempted_at`,
  `total_bytes`.

### `SandboxExecutionResultStore.stats()`
- Returns aggregate stats by scanning all loadable records and computing
  disk usage. Corrupt/unparseable files are silently skipped.

### `SandboxExecutionResultStore.plan_prune(keep_latest, status)`
- Dry-run preview: returns the list of records that would be deleted by
  `prune()` with the same parameters. No filesystem changes.
- `keep_latest`: keep newest N records (None = no limit, keep all).
- `status`: optional filter ("completed", "blocked_claim").
- Raises `ValueError` on invalid parameters.

### `SandboxExecutionResultStore.prune(keep_latest, status)`
- Actually deletes result files. Safety guards:
  - Skips symlinks (`path.is_symlink()`).
  - Skips non-regular files (`path.is_file()`).
  - Only deletes valid, loadable `.json` records under
    `.sac/sandbox_executions/`.
  - Catches `OSError` and continues.
- Returns count of deleted files.

### CLI Changes
- `sac sandbox executions stats` — aggregate statistics table.
- `sac sandbox executions prune --keep-latest N --dry-run` — preview deletion
  candidates as a table.
- `sac sandbox executions prune --keep-latest N --yes` — confirmed deletion.
- `--status completed|blocked_claim` filter on prune.
- `sac sandbox executions` (no subcommand) continues to work as the list
  command.

## Security
- Prune never deletes files outside `.sac/sandbox_executions/`.
- Symlinks are explicitly skipped.
- Corrupt/unparseable files are never deleted.
- No sensitive content is exposed in stats or prune output.
- Both prune modes require explicit `--dry-run` or `--yes`.

## Backward Compatibility
- `sac sandbox executions` with filters unchanged (now default callback of
  `executions_app` group).
- All existing store methods (`filter_by`, `load`, `latest`, `list_all`)
  unchanged.
