## Problem
The interactive loop can maintain state and route typed intents, but real model
output still needs a narrow contract before it can safely drive planning and
tool selection.

## Goals
- [x] Add discriminated response schemas for `answer`, `plan`, `tool_intent`,
  `patch`, `stop_for_user`, and `error`.
- [x] Add a parser that accepts JSON objects and fenced JSON while failing
  closed on malformed or unknown response types.
- [x] Extend `LLMClient` with `plan(...)` and `choose_tool(...)`.
- [x] Implement deterministic mock plan and tool-choice responses.
- [x] Add OpenAI-compatible structured plan/tool-choice calls without making
  tests require a real API key.
- [x] Route model-produced tool intents through the existing `ToolIntentRouter`.
