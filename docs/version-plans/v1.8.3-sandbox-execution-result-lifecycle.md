# v1.8.3: Sandbox Execution Result Lifecycle

## Status
Planned.

## Base
- Depends on: `v1.8.2`
- Baseline context: `.claude/skills/current/SKILL.md`
- Runtime invariants: `.claude/skills/shared/core-runtime.md`
- New tag: `v1.8.3`

## Problem
v1.8.2 leaves a completed or claim-blocked execution's pending proposal file
on disk indefinitely. There is no persistent record of what happened, and
the pending file can be re-read as if execution were still pending.
Blocked preflight also has unclear lifecycle semantics.

## Goals
- [x] Successful or failed execution clears the pending proposal so it
      cannot appear as retriable.
- [x] New `SandboxExecutionResultStore` persists an immutable result record
      per execution attempt under `.sac/sandbox_executions/<proposal_id>.json`.
- [x] Result records redact/truncate output — no full command stored, no env
      values in structured fields, stdout/stderr truncated at 2000 chars.
- [x] Claim failure writes a `blocked_claim` result record, then clears pending.
- [x] Preflight blocked preserves the pending proposal so the user can fix
      configuration and retry. No result record is written.
- [x] CLI commands `sac sandbox executions` and `sac sandbox last-execution`
      let users inspect past results.
- [x] v1.8.2 safety semantics preserved: atomic claim still happens before
      execution, blocked preflight does not consume approval.

## Requirements
- [x] `SandboxExecutionResultRecord` (frozen dataclass) with fields:
      proposal_id, attempted_at, backend, executed, exit_code, duration_ms,
      status, message, command_hash_prefix, command_head, stdout_preview,
      stderr_preview, stdout_length, stderr_length.
- [x] `SandboxExecutionResultStore.save()` writes one record per proposal_id.
- [x] `SandboxExecutionResultStore.load()` / `list_all()` / `latest()`
      for retrieval.
- [x] `execute_pending()` writes a result record and calls
      `self.store.discard_pending()` after claim failure or shell completion.
- [x] Preflight-blocked path unchanged: pending file preserved, no result
      record written.
- [x] `SanboxExecutionResultStore._truncate()` caps output at
      `MAX_RESULT_PREVIEW_LENGTH` (2000).

## Backward Compatibility
- All existing CLI commands and signatures preserved.
- Preflight, approval, claim, and audit behavior unchanged except that
  pending is now cleared after execution/claim-failure.
- `claim_for_execution()` unchanged.
- Existing test suites updated where they assumed pending survived execution.

## Acceptance Criteria
- [x] Successful execution writes `completed` result record and clears pending.
- [x] Non-zero exit writes `completed` result record and clears pending.
- [x] Claim failure writes `blocked_claim` result record and clears pending.
- [x] Preflight blocked preserves pending and does not consume approval.
- [x] Result record does not contain env values in structured fields.
- [x] Result record does not contain the full command argv.
- [x] stdout/stderr are truncated when exceeding MAX_RESULT_PREVIEW_LENGTH.
- [x] `SandboxExecutionResultStore.list_all()` returns newest-first records.
- [x] `SandboxExecutionResultStore.latest()` returns the most recent record.
- [x] `sac sandbox executions` lists records.
- [x] `sac sandbox last-execution` shows the latest record with previews.
- [x] Manually discarding pending does not write a result record.
- [x] Claim failure does not invoke subprocess.
