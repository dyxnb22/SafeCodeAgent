# v1.8.6: Sandbox Execution Result Maintenance

## Status
Implemented and tagged as `v1.8.6`.

Post-tag maintenance note: a follow-up hardening patch keeps `prune()` tied to
the actual source file discovered during store scans, so a malformed-but-valid
record cannot claim another record's `proposal_id` and cause the wrong file to
be deleted.

- Depends on: `v1.8.5`

## Problem
As execution records accumulate in `.sac/sandbox_executions/`, users have no
way to view aggregate statistics or safely clean up old records. Manual file
deletion is error-prone and could remove non-result files.

## Goals
- [x] `SandboxExecutionResultStats` frozen dataclass with aggregate counts
      and disk usage.
- [x] `stats()` method on `SandboxExecutionResultStore`.
- [x] `plan_prune(keep_latest, status)` for dry-run preview — returns the list
      of records that would be deleted, without touching disk.
- [x] `prune(keep_latest, status)` for safe deletion — only deletes valid,
      loadable `.json` result files. Skips symlinks, non-files, corrupt JSON.
- [x] CLI: `sac sandbox executions stats` — table of aggregate metrics.
- [x] CLI: `sac sandbox executions prune --keep-latest N --dry-run` — shows
      candidates without deleting.
- [x] CLI: `sac sandbox executions prune --keep-latest N --yes` — confirmed
      deletion.
- [x] CLI: `--status completed|blocked_claim` filter on prune.
- [x] CLI: prune requires `--dry-run` or `--yes`; neither produces error
      exit 1.
- [x] All destructive operations are explicit and auditable.

## Requirements
- [x] `SandboxExecutionResultStats(total_records, completed_count,
      blocked_claim_count, oldest_attempted_at, newest_attempted_at,
      total_bytes)`.
- [x] `stats()` uses `_disk_usage()` which only counts `.json` regular files,
      skipping symlinks and catching OSError.
- [x] `plan_prune()` sorts newest-first, determines deletion candidates,
      returns them without side effects.
- [x] `prune()` validates params, uses the same pruning plan as
      `plan_prune()`, then safely deletes only matched source files via
      `path.unlink()` with symlink + is_file guards.
- [x] `_validate_prune_params()` rejects `keep_latest < 0` and unknown
      status values.
- [x] CLI uses `executions_app` Typer sub-app with `invoke_without_command`
      callback so the list command stays the default action.

## Backward Compatibility
- `sac sandbox executions` still works as the list command (now the default
  callback of the `executions_app` group).
- `sac sandbox executions --status --backend --proposal-id --limit --sort`
  all work unchanged.
- Existing `filter_by()`, `load()`, `latest()`, `list_all()` unchanged.

## Acceptance Criteria
- [x] Stats on empty store returns all-zero metrics.
- [x] Stats correctly distinguishes completed vs blocked_claim counts.
- [x] Stats reports non-zero byte count when records exist.
- [x] `plan_prune()` returns candidates but does not delete.
- [x] `prune(keep_latest=2)` keeps newest 2, deletes older 3.
- [x] `prune(status="blocked_claim")` only deletes matching status.
- [x] `keep_latest=-1` raises ValueError.
- [x] Unknown status raises ValueError.
- [x] Symlinks in executions dir are never deleted.
- [x] Corrupt JSON is not crashed on by stats or deleted by prune.
- [x] A valid record with a mismatched `proposal_id` cannot cause `prune()`
      to delete another record's file.
- [x] `keep_latest >= len(records)` returns empty candidate list.
- [x] `keep_latest=0` deletes all valid records.
- [x] CLI prune without `--dry-run` or `--yes` exits 1.
- [x] CLI stats table renders correctly.
