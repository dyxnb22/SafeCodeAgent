# v2.2.1 — Model Tool Call Adapter

## Goal
Convert structured LLM tool intents into validated runtime tool calls by consulting
the v2.2.0 `ToolRegistry` before any routing decision is made.

## Requirements

### New module: `src/safecode/tools/adapter.py`
- `AdapterError(ValueError)` — explicit, testable error type for all validation failures.
- `ToolCallValidationResult` — frozen Pydantic model carrying `tool_name`, `spec`,
  `resolved_args`, `requires_approval`, `risk`, `permission_category`, `audit_event`.
- `ToolCallAdapter` — validates tool name, required args, and arg types against
  `ToolRegistry`; returns `ToolCallValidationResult` or raises `AdapterError`.
  - `validate(tool_name, args)` — full validation; raises for unknown tool, missing
    required arg, or type mismatch.
  - `lookup(tool_name)` — name-only lookup; raises `AdapterError` for unknown names.
- No tool execution, no LLM calls, no file writes.

### Updated: `src/safecode/agent/tools.py`
- `ToolIntentRouter` consults `ToolCallAdapter.lookup()` for every intent type it
  routes, using a `_REGISTRY_NAMES` mapping from intent type to registry tool name.
- Approval requirement is derived from `ToolSpec.requires_human_approval` (registry
  authoritative) OR the existing `requires_approval` flag on the intent — whichever
  is more restrictive.
- MCP intents default to `mcp.propose_write` in registry lookup (conservative).
- Unknown registry names raise `ValueError` (fail closed), not a silent fallback.
- All existing public route strings and `RoutedToolIntent` fields remain unchanged.

### Adapter validation rules
- **Unknown tool name** → `AdapterError: Unknown tool: 'name'`
- **Missing required arg** → `AdapterError: Missing required argument 'arg' for tool 'name'`
- **Invalid arg type** → `AdapterError: Argument 'arg' for tool 'name': expected TYPE, got ACTUAL`
- **Extra args** → tolerated (not declared in schema = not validated)
- **High-risk / WRITE / SHELL tools** → `requires_approval=True` always (from registry)
- **Read-only / AUDIT tools** → `requires_approval=False` when registry says so

### Safety constraints
- Do not execute any tool from the registry.
- Do not call the LLM.
- Do not add new shell execution paths.
- Do not add MCP write execution.
- Do not bypass existing approval, patch, sandbox, audit, redaction, or command-policy gates.
- Project config must not weaken user-level safety policy.

## Tests (`tests/test_tool_call_adapter.py`)
- valid known tool adapts successfully
- unknown tool is rejected with `AdapterError`
- missing required arg is rejected
- each common arg type mismatch is rejected (`str`, `int`, `bool`, `path`, `dict`, `list`)
- WRITE/SHELL/high-risk tools preserve `requires_approval=True`
- read-only tools return `requires_approval=False`
- router backward compat: all existing valid intents still route correctly
- adapter has no subprocess, file-write, or LLM side effects
