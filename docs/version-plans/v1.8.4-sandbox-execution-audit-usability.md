# v1.8.4: Sandbox Execution Audit & Usability

## Status
Planned.

- New tag: `v1.8.4`

## Problem
v1.8.3 result records have no schema version marker, making future migration
risky. The `load()`/`list_all()` methods fail on records with extra JSON
fields, breaking forward compatibility. The `sac sandbox executions` CLI
lists everything without filters, and `sac sandbox status` doesn't mention
execution results at all.

## Goals
- [x] Add `_schema_version: "v1"` to every saved result record JSON so
      future versions can detect and migrate old records.
- [x] Make `load()` and `list_all()` tolerant of unknown JSON keys
      (forward-compatible), but still require all known required fields.
- [x] Add `filter_by(backend, status, proposal_id_substr)` on
      `SandboxExecutionResultStore` for programmatic and CLI use.
- [x] Add `--status`, `--backend`, `--proposal-id` options to
      `sac sandbox executions`.
- [x] Add execution result summary (counts + latest) to `sac sandbox status`.
- [x] Preserve all v1.8.3 safety semantics: redaction, truncation, pending
      lifecycle unchanged.

## Requirements
- [x] `RECORD_SCHEMA_VERSION = "v1"` constant in `execution.py`.
- [x] `save()` writes `_schema_version` as the first key.
- [x] `_filter_record_data()` strips unknown fields before dataclass
      construction; returns `None` when required keys are missing.
- [x] `load()` and `list_all()` use `_filter_record_data()`.
- [x] `filter_by()` accepts optional `backend`, `status`, `proposal_id_substr`.
- [x] CLI `--status`, `--backend`, `--proposal-id` pass through to `filter_by()`.
- [x] `sac sandbox status` appends an "Execution Results" panel when records
      exist.

## Backward Compatibility
- Existing result records without `_schema_version` load fine — the field is
  for writing only, `_filter_record_data()` ignores it on read.
- Extra fields in future result records won't break loading.
- All existing CLI commands and signatures preserved.
- `SandboxExecutionResultRecord` frozen dataclass unchanged (schema version
  is JSON-only).
