## Problem
v1.8.8 hardened sandbox execution result persistence with same-directory
temporary files and `os.replace(...)`. Pending sandbox execution proposals
still used direct `Path.write_text(...)` when first created, leaving the
proposal store less consistent than the result store.

## Goals
- [x] Write pending sandbox execution proposals through a same-directory
      temporary file.
- [x] Atomically publish the proposal with `os.replace(...)`.
- [x] Replace a broken same-name symlink rather than following it.
- [x] Clean up temporary files if publishing fails.
- [x] Preserve proposal schema and lifecycle behavior.

## Requirements
- [x] `SandboxExecutionProposalStore._write(...)` writes to
      `.sac/.pending_sandbox_execution.json.<uuid>.tmp`.
- [x] `_write(...)` uses `os.replace(temp_path, pending_path)`.
- [x] The `finally` block removes leftover temp files when possible.
- [x] Broken symlink targets outside the project are not created or modified.
- [x] Failed replace operations do not leave `*.tmp` files in `.sac/`.
