## Problem
v1.8.6 added safe result-store pruning and documented destructive
maintenance as explicit and auditable. The confirmed CLI prune path deleted
records safely, but it only printed a terminal message and did not write an
audit event.

## Goals
- [x] `sac sandbox executions prune --yes` writes a durable audit event after
      confirmed deletion.
- [x] `sac sandbox executions prune --dry-run` remains preview-only and does
      not write a destructive maintenance event.
- [x] Audit metadata records the pruning parameters and counts without
      retaining stdout/stderr, full commands, or sensitive values.
- [x] Existing prune deletion safety behavior remains unchanged.

## Requirements
- [x] Audit event type: `sandbox_execution_results_pruned`.
- [x] Event status: `success`.
- [x] Metadata includes `keep_latest`, `status_filter`, `candidate_count`,
      `deleted_count`, `remaining_count`, `proposal_ids`, and
      `proposal_ids_truncated`.
- [x] `proposal_ids` is capped to the first 20 candidates.
- [x] Dry-run does not create `sandbox_execution_results_pruned`.
